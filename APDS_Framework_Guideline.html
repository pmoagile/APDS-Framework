Agile Project Delivery System (APDS) Framework

Interactive Guideline — Inputs, Processes, Outputs, Indicators & Decisions

What is APDS?

The Agile Project Delivery System (APDS) is a supplementary measurement and decision-making framework for Scrum initiatives. It tracks four core indicators — Cumulative Delivery Pace (CDP), Resource Efficiency (RE), Scope Stability (SS), and Collaboration & Satisfaction (CS) — each classified into Green, Amber, or Red zones. These feed into a Project Score that maps to four decision levels: Proceed, Watch, Review, or Escalate.

Core Capabilities

  • Quantifiable Confidence — Every indicator scores 0–1, eliminating guesswork
  • Data-Driven Decisions — Project Score maps to actionable levels
  • Early Warning System — Amber/Red zones flag issues before crises
  • Forecast Accuracy — Predict total sprints needed via FCP
  • Governance Traceability — Auditable, repeatable decisions

Who Should Use It

Project Managers, Product Owners, Scrum Masters, and team leads who want to back decisions with measurable data — without interrupting the team's daily Scrum workflow.

APDS Inputs

The framework requires the following foundational data elements, established during project setup and maintained throughout the lifecycle.

Work Units (WU)

Each requirement (feature, user story, task) tracked via three elements:

  • Fibonacci Estimate — 1, 2, 3, 5, 8, 13… (fixed unless scope changes)
  • WPL Rung — R1–R5 tracking state
  • Weighted Progress — Fib × Rung completion %

Work Progression Ladder (WPL)

RungDescriptionCompletion
R1: DraftWork identified and assigned0%
R2: AssemblyWork in progress, components built25%
R3: ValidationReady for review/testing60%
R4: CertifiedPassed validation, approved85%
R5: AgreedAccepted by PO/customer100%

Planning Parameters

  • Total Project Budget — Distributed across sprints
  • Total Number of Sprints — From release plan
  • Original WU Count — Baseline scope
  • MoSCoW Prioritization — Must/Should/Could/Won't Have

Feedback Data (per Sprint)

  • Team Collaboration ratings (1–5, 3 statements)
  • Stakeholder Satisfaction ratings (1–5, 2 statements)
  • Communication ratings (1–5, 2 statements)
  • Actual Sprint Cost (SC)

Sprint Planning Integration

The PO presents previous sprint's indicators, Project Score, and FCP. The team uses this data to determine sprint capacity and commitment:

  • If CDP is Critical → consider lighter load
  • If RE is Critical → investigate resourcing
  • If SS is Creeping/Unstable → factor scope growth
  • If CS is Damaged/Broken → address collaboration first

The Four Core Indicators

Each indicator scored 0–100% and classified into Green (90–100%), Amber (80–89%), or Red (0–79%).

Cumulative Delivery Pace (CDP)

Question: Is delivery on track?

Formula: CDP = APcum / PPcum

What it measures: Compares cumulative actual progress against cumulative planned progress across all sprints.

Resource Efficiency (RE)

Question: Is cost within budget?

Formula: RE = SB / SC (capped at 100%)

What it measures: Sprint Budget vs. actual Sprint Cost. Cross-check against CDP for accuracy.

Scope Stability (SS)

Question: Is scope stable?

Formula: SS = 1 − (|Current − Original| / Max(Current, Original))

What it measures: Variance between original and current WU count.

Collaboration & Satisfaction (CS)

Question: Is the team healthy?

Formula: CS = (Collab × 0.4) + (Sat × 0.4) + (Comm × 0.2)

What it measures: Blends team collaboration, stakeholder satisfaction, and communication quality.

Project Score (PS)

Formula: PS = (CDP + RE + SS + CS) / 4

The single most important health indicator. Maps directly to decision levels: Proceed, Watch, Review, Escalate.

Forecast Completion Point (FCP)

Formula: FCP = Total Sprints + ((PPcum − APcum) × Total Sprints)

Predicts total sprints required based on CDP gap analysis.

1. Planned Progress (PP)

PPsprint = Σ(WU Fib × 100%) / Σ(WU Fib) = 1.0 (100%)

PP is always 100% for committed scope. Recalculated mid-sprint if scope changes.

PPcum = (100% / Total Sprints) × Current Sprint Number

2. Actual Progress (AP)

AP = Σ(WU Fib × Rung Completion %) / Σ(WU Fib)

Weighted completion based on WPL status. Updated during Daily Scrum.

MoSCoW Prioritization with APDS Scoring

Each Work Unit is scored 1–100 based on business value, strategic alignment, and stakeholder impact.

CategoryScore Range
Must Have80–100
Should Have50–79
Could Have20–49
Won't HaveNot scored (out of scope)

MoSCoW Distribution Ratio (MDR)

MDR = Must Have count / Total requirements

Zone Classification:

Balanced
30–60%
Healthy distribution. Room to absorb changes.
Top-Heavy
> 60%
Rigid backlog. No safety buffer.
Thin
< 30%
Weak core. Unreliable MVP.

MoSCoW Value Density (MVD)

MVD = Total Value Score / Total requirements

Zone Classification:

Healthy
≥ 50
High-value backlog. Sprint capacity well spent.
Low-Value
< 50
Low-value work dominates. Critical items undelivered.

MoSCoW Stability Index (MSI)

MSI = 1 − (Changed / Total)

Measures priority stability. 100% means no priorities changed.

MoSCoW Cross-Indicator Impact

  • MDR > 60% → SS drops — Top-heavy backlog causes scope creep, lowering Scope Stability
  • Frequent MoSCoW re-categorization → CS drops — Stakeholder disagreement undermines collaboration and satisfaction
  • MVD < 50 → Financial risk — Low-value backlog signals value-for-money problem

Forecasting Engine

The Forecast Completion Point (FCP) estimates total sprints required based on the delivery pace gap.

FCP = Total Sprints + ((PPcum − APcum) × Total Sprints)

Example: 10 total sprints, at Sprint 5 (PPcum=50%), APcum=30% → FCP = 10 + ((50%−30%)×10) = 12 sprints

Recovery Plan Deliverable

When PS triggers Review or Escalate, the problem-solving workshop produces a structured recovery plan containing:

  • Specific Corrective Actions — Concrete remediation tasks
  • Review Date — Firm deadline for evaluating intervention effectiveness

Workshop Sequence: Sprint Retrospective → Problem-Solving Workshop → Sprint Planning

APDS Checklists

Click items to mark them complete.

Project Setup Checklist

  • Create Project Charter / Vision / BRD
  • Log all requirements as candidate Work Units
  • Apply MoSCoW prioritization to all Work Units
  • Map release plan with sprint count
  • Set total project budget
  • Assign Fibonacci estimates to Work Units
  • Record dependencies in RAID log
  • Establish original WU count baseline

Per-Sprint Data Collection

  • Update WPL status for all active Work Units (Daily Scrum)
  • Collect collaboration ratings (3 statements, 1–5)
  • Collect communication ratings (2 statements, 1–5)
  • Collect stakeholder satisfaction ratings (2 statements, 1–5)
  • Record actual Sprint Cost (SC)
  • Recalculate WU count if scope changed

Sprint Review & Retrospective

  • Calculate AP, PP, CDP, RE, SS, CS
  • Calculate Project Score (PS)
  • Determine decision level (Proceed/Watch/Review/Escalate)
  • Calculate Forecast Completion Point (FCP)
  • Review MoSCoW distribution (MDR, MVD, MSI)
  • Present indicators and PS to team/stakeholders
  • If PS < 90%: Schedule problem-solving workshop

Problem-Solving Workshop

  • Identify root causes of underperforming indicators
  • Define specific corrective actions
  • Assign item owners
  • Set firm review date
  • Feed recovery plan into Sprint Planning
